Writing completions

Rush completions are structured rules plus optional Rush functions. Small completions can live entirely in .rush files. Larger generated completions can put the static command grammar in a JSON completion manifest and keep dynamic providers in a companion .rush file.

The same engine consumes both formats. It parses the current command line into semantic positions, evaluates the matching rules, runs only the providers that apply, filters and ranks candidates, then applies a shell-quoted edit.

Put a completion file at one of these paths:

$XDG_DATA_HOME/rush/completions/COMMAND.rush
$HOME/.local/share/rush/completions/COMMAND.rush
$XDG_CONFIG_HOME/rush/completions/COMMAND.rush
$HOME/.config/rush/completions/COMMAND.rush

Manifest-backed completions use the same locations with .json:

$XDG_DATA_HOME/rush/completions/COMMAND.json
$HOME/.local/share/rush/completions/COMMAND.json
$XDG_CONFIG_HOME/rush/completions/COMMAND.json
$HOME/.config/rush/completions/COMMAND.json

System completions are loaded from:

$XDG_DATA_DIRS/rush/completions/COMMAND.rush
/usr/local/share/rush/completions/COMMAND.rush
/usr/share/rush/completions/COMMAND.rush

System manifests are loaded from the same directories as COMMAND.json. Rush loads completion data lazily when completing COMMAND for the first time. If a manifest and .rush file are colocated, the manifest owns the static rules and the .rush file is treated as a provider-only companion that is sourced lazily when a manifest function provider needs it.

Engine lifecycle

A completion request follows this lifecycle:

parse input
  -> build semantic context
  -> load COMMAND.rush or COMMAND.json
  -> select matching structured rules
  -> run matching dynamic providers
  -> add default/builtin fallbacks when no explicit rule applies
  -> fuzzy-filter, sort, deduplicate, and rank candidates
  -> compute a quoted insertion edit

Providers run with a snapshot of shell state from the request. Variables, aliases, functions, and registered completion rules are available to the provider, but mutations to shell state are discarded after completion. Filesystem and external command side effects are not sandboxed, so providers should be read-only queries.

Interactive providers run asynchronously so the editor stays responsive. A newer completion request supersedes older results; Rush discards stale candidates and asks in-flight external commands to terminate.

Static completions

Use complete to register structured rules. The first argument is a command pattern: the root command plus optional subcommand path. Quote multi-word patterns.

complete git --subcommand commit --description 'record changes'
complete git --subcommand checkout --description 'switch branches'

complete git --option --long help --description 'show help'
complete git --option --short C --value-name path --description 'run as if in path'

complete 'git commit' --option --long amend --description 'amend previous commit'
complete 'git commit' --option --short m --long message --value-name text --description 'use text as message'

Rule forms:

complete PATTERN --subcommand NAME [--description TEXT]
complete PATTERN --option [--short C] [--long NAME] [--value-name NAME]
                 [--exclusive-group GROUP] [--repeatable] [--terminates-options]
                 [--no-space] [--description TEXT]
complete PATTERN --subcommands --function FUNC
complete PATTERN --options --function FUNC
complete PATTERN --argument --function FUNC [argument selectors] [value grammar]
complete PATTERN --option-value (--long NAME | --short C) --function FUNC [value grammar]

Options are non-repeatable by default. After a known option is already present, Rush suppresses that option and reports repeated uses in completion diagnostics. Add --repeatable for options that may occur more than once. Use --exclusive-group for mutually exclusive options; once any option in the group is present, the others are suppressed. A bare -- always ends option parsing; use --terminates-options for command-specific option terminators such as sudo ---like pass-through flags.

Dynamic completions

Use a provider function when candidates depend on the current machine, repository, environment, or parsed command line. The function runs only during completion and emits candidates through the completion builtin.

__rush_complete_git_branches() {
  completion candidate main --description 'branch'
  completion candidate next --description 'branch'
}

complete 'git checkout' --argument --function __rush_complete_git_branches

Dynamic rules are semantic: --subcommands providers run only at a subcommand slot, --options providers only at an option slot, --argument providers only at matching operand states, and --option-value providers only while completing the named option value.

Use completion candidates when an external command already returns one value per line:

__rush_complete_brew_formulae() {
  brew search 2>/dev/null | completion candidates --kind plain --description formula
}

complete 'brew install' --argument --function __rush_complete_brew_formulae

Candidate fields

completion candidate VALUE [--display TEXT] [--description TEXT] [--kind KIND] [--no-space]
completion candidates [--description TEXT] [--kind KIND] [--no-space]  # reads newline-separated values on stdin

Kinds:

command builtin function file directory variable option subcommand plain

VALUE is the canonical value to insert. DISPLAY is only shown in the menu and also participates in matching. DESCRIPTION is shown next to the menu label. --no-space prevents the automatic trailing space after insertion.

Structured options and option values

Prefer static complete --option rules for known options, and dynamic completion option when the option set itself is computed. Both forms create structured option candidates.

__rush_complete_tool_options() {
  completion option --long verbose --description 'show verbose output'
  completion option --long include --repeatable --description 'include a path'
  completion option --short o --long output --argument file --description 'write to file'
}

complete tool --options --function __rush_complete_tool_options

An option value provider is separate from the option declaration. Declare the option with --value-name or completion option --argument, then add a scoped value provider. Rush detects detached values (--color auto, -m text), attached long values (--color=auto), and attached short values when a short option that takes a value is followed by text (-Dkey=value).

__rush_complete_color_modes() {
  completion candidate auto
  completion candidate always
  completion candidate never
}

complete tool --option --long color --value-name when --description 'when to use color'
complete tool --option-value --long color --function __rush_complete_color_modes

Argument states

Argument rules can be plain providers, fixed indexes, named state machines, or repeatable trailing states. Option values are skipped when Rush computes operand indexes.

__kubectl_types() { completion candidate pods; completion candidate services; }
__kubectl_names() { completion candidate nginx; completion candidate api; }

complete kubectl --subcommand get
complete 'kubectl get' --option --long namespace --value-name name
complete 'kubectl get' --argument --state type --function __kubectl_types
complete 'kubectl get' --argument --state name --after-state type --repeatable --function __kubectl_names

Selectors:

--index N          # provider applies at zero-based operand index N
--state NAME       # provider contributes to a named active state
--after-state NAME # state applies after the previous operand had NAME
--after VALUE      # state applies after the previous operand value equals VALUE
--repeatable      # state repeats from its index or transition onward

Value grammars

Value grammars tell Rush to replace only the active segment inside a structured value and expose that segment to providers. The .rush DSL currently supports list values and key/value pairs.

__tool_modes() {
  completion candidate slow --description "$rush_completion_value_position:$rush_completion_value_segment:$rush_completion_value_separator"
}

complete tool --option --long mode --value-name mode
complete tool --option-value --long mode --function __tool_modes --list-separator ,

Completing tool --mode=fast,sl runs the provider with rush_completion_value_segment=sl and replaces only sl.

__tool_defines() {
  if test "$rush_completion_value_position" = value; then
    completion candidate one
  else
    completion candidate alpha
  fi
}

complete tool --option --short D --long define --value-name pair
complete tool --option-value --long define --function __tool_defines --key-value-separator =
complete tool --option-value --short D --function __tool_defines --key-value-separator =

Completing tool --define alpha=o exposes value-position=value, value-key=alpha, and replaces only o. Completing tool --define al exposes value-position=key.

Context

Completion functions can inspect Rush's parsed view of the current command line. Prefer the lowercase rush_completion_* variables for scalar context values; they avoid command substitutions in hot completion providers.

$rush_completion_prefix              # text being completed
$rush_completion_command             # root command
$rush_completion_command_path        # root plus matched subcommand path
$rush_completion_previous            # previous word
$rush_completion_position            # command, subcommand, option, option_value, argument
$rush_completion_argument_index      # zero-based active argument index
$rush_completion_argument_state      # active structured argument state, if any
$rush_completion_options_terminated  # true after -- or a terminator option
$rush_completion_option_name         # active option name, in option_value position
$rush_completion_option_spelling     # active spelling, such as --color or -o
$rush_completion_value_segment       # active structured value segment
$rush_completion_value_separator     # active structured value separator
$rush_completion_value_position      # key or value in structured values
$rush_completion_value_key           # key for structured key/value completions

Use parameterized helpers for queries that need arguments or return lists. completion option-present exits successfully when the option is present and unsuccessfully otherwise, so it can be used directly in if.

completion prefix          # text being completed
completion command         # root command
completion argument-index  # zero-based argument index
completion operand-index   # alias for active argument index
completion operands        # completed operands, one per line
completion previous        # previous word
completion position        # command, subcommand, option, option_value, argument
completion option-name     # active option name, in option_value position
completion option-spelling # active spelling, such as --color or -o
completion value-segment   # active structured value segment
completion value-separator # active structured value separator
completion value-position  # item, key, or value
completion value-key       # key for key/value completions
completion option-present --long NAME
completion option-values --long NAME
completion options-terminated
if completion option-present --long staged; then
  completion candidate staged-file --kind file
fi

Context queries use Rush's semantic completion analysis. Providers can branch on parsed option presence, attached or detached option values, completed operands, named argument states, structured value segments, and whether a bare -- or terminator option ended option parsing.

Matcher policy

Rush owns filtering. Providers should emit all plausible candidates for the active semantic slot and let the engine filter against the current prefix.

The engine-default matcher is case-insensitive, fuzzy, treats hyphen and underscore as equivalent, and matches full path text. Matches are ranked exact, then prefix, then fuzzy. Both VALUE and DISPLAY can match. The active matcher policy is visible in rush complete trace.

Because filtering is engine-owned, completion authors should not pre-filter unless an external query requires it for performance. Pre-filtering too aggressively can hide fuzzy matches that Rush would otherwise surface.

Quoting and insertion

Emit candidate values unescaped. Rush decodes the current shell word for matching, then encodes the replacement for the quoting context at the edit span.

completion candidate 'feature branch'  # inserted as feature\ branch when unquoted
completion candidate 'cost $5'        # $ is escaped outside single quotes

Inside single or double quotes, Rush preserves the quote style and escapes only what that context requires. If a completed quoted value appends a space, Rush closes the quote before adding the space. Directory candidates from path helpers append / and do not add a trailing space; use --no-space for other candidates that should keep the cursor adjacent to the inserted text.

Helper completions

Rush can add common candidate sets for you.

completion files [--prefix PREFIX] [--extension EXT]
completion directories [--prefix PREFIX] [--append-slash]
completion executables [--prefix PREFIX]
completion variables [--prefix PREFIX]

If --prefix is omitted, Rush uses the current completion prefix.

Trace and debug workflow

Use rush complete trace INPUT to inspect what the engine sees and why candidates appear or disappear.

$ rush complete trace 'git diff --cached ma'

The text trace includes:

context:             # provider-facing values: prefix, position, option value, value segment
matcher-policy:      # active filtering policy
semantic:            # parsed command path, options, operands, argument state
rules:               # matching rules and whether they came from .rush or a manifest
suppressed-options:  # repeat/exclusive-group suppression reasons
candidates:          # candidate edit spans, insertion text, match rank, ranking score
provider-diagnostics:# missing functions, non-zero provider exits, stderr
application:         # edit Rush would apply, or ambiguous/none

For machine-readable output, use the current JSON debug form:

$ rush complete --debug-json 'git diff --cached ma'

A practical authoring loop is: add the smallest static rule, run rush complete trace for the target cursor position, confirm semantic.position and argument-state, add a provider only if static candidates are insufficient, then check provider diagnostics and candidate replacement spans. Use rush complete validate path/to/file.rush to parse-check .rush completion scripts.

Complete example

File: $HOME/.local/share/rush/completions/example.rush

complete example --subcommand run --description 'run a target'
complete example --subcommand clean --description 'remove build output'
complete example --option --long help --description 'show help'

__rush_complete_example_targets() {
  completion candidate app --kind plain
  completion candidate tests --kind plain
}

complete 'example run' --argument --function __rush_complete_example_targets

Trace it with:

$ rush complete trace 'example run ap'

Status

The structured completion engine, .rush DSL, manifest v1 loader, and trace workflow are implemented. The API is still young; prefer documented rule forms and avoid relying on undocumented trace fields.
